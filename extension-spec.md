# FIA-X行业扩展接口规范 v1.0

## 一、扩展原则
✅ **完全复用核心能力**：所有行业扩展包100%复用FIA-X核心层的三大认知支柱、十步流程、多智能体引擎、评估体系、数学模型，不需要重复开发核心功能  
✅ **仅补充行业特性**：行业扩展包仅包含本行业的专属内容，通用内容统一放在核心层  
✅ **接口标准化**：所有行业扩展包遵循统一的接口规范，便于管理和扩展  
✅ **向后兼容**：核心层升级不影响已有的行业扩展包，所有扩展包自动继承核心层的新能力

---

## 二、扩展包结构规范
所有行业扩展包必须遵循以下目录结构：
```
fia-industry-xxx/    # xxx为行业名称，小写，用连字符分隔，如fia-industry-medical、fia-industry-new-energy
├── SKILL.md         # 行业扩展包核心文件（必填）
├── references/      # 行业专属参考资料（可选）
│   ├── xxx.md
│   └── ...
├── scripts/         # 行业专属工具脚本（可选）
│   ├── xxx.py
│   └── ...
└── assets/          # 行业专属资源文件（可选）
    ├── templates/
    ├── cases/
    └── ...
```

---

## 三、SKILL.md核心接口规范
行业扩展包的SKILL.md必须包含以下标准化字段：

### 1. 头部元数据（必填）
```yaml
---
name: fia-industry-xxx  # 扩展包名称，必须与目录名一致
description: 一句话描述该行业扩展包的功能和适用场景，必须包含"基于FIA-X通用核心层"字样
industry: 行业名称，如"医疗大模型"、"新能源汽车"、"智能制造"
version: 版本号，如v1.0
---
```

### 2. 核心定位（必填）
- 明确说明与FIA-X核心层的关系："本扩展包100%继承FIA-X通用核心层的所有能力，仅补充[行业名称]的专属特性"
- 适用场景：列出该行业扩展包适用的具体场景

### 3. 行业专属规则（必填）
列出该行业不可突破的核心原则、监管要求、底线约束。

示例（医疗行业）：
```
## 行业专属规则
### 1. 医疗安全优先原则
医疗的核心是"治病救人"，任何错误都可能直接导致人身伤害甚至死亡，容错率为0。
### 2. 证据驱动原则
所有诊疗决策必须有明确的医学证据支撑，遵循循证医学原则。
```

### 4. 流程适配接口（可选但推荐）
基于FIA-X核心十步流程，说明每个步骤在本行业的定制化调整和新增环节。
格式要求：
```
## 行业定制化流程
在FIA-X通用十步流程基础上，增加以下行业专属环节：

### Step 1：问题本质拆解（行业定制）
✅ 通用环节：（保留核心层的通用说明）
➕ 行业新增：（说明本行业在该步骤的新增要求）
- 新增1：xxx
- 新增2：xxx
📌 输出物增加：《xxx报告》
```

### 5. 行业专属资源（可选）
列出该扩展包提供的专属参考资料、工具脚本、案例库等资源。
格式要求：
```
## 行业专属资源
### 参考资料
| 资料名称 | 功能描述 |
|----------|----------|
| `references/xxx.md` | 功能说明 |

### 工具脚本
| 脚本名称 | 功能描述 | 调用环节 |
|----------|----------|----------|
| `scripts/xxx.py` | 功能说明 | 调用的步骤编号 |
```

### 6. 行业成功案例（可选但推荐）
提供1-2个本行业应用FIA-X方法论的成功案例，说明应用效果。

## 四、扩展开发流程
### 1. 扩展包创建步骤
1. 复制模板目录：`cp -r templates/fia-industry-template fia-industry-xxx`
2. 修改SKILL.md中的元数据和内容
3. 补充行业专属的参考资料、工具脚本、资源文件
4. 按照规范测试兼容性，确保不修改核心层任何内容
5. 提交审核，通过后纳入行业扩展生态

### 2. 兼容性要求
- 不得修改核心层的任何文件
- 新增的内容不得与核心层的命名空间冲突
- 必须支持核心层的版本升级，核心层升级时扩展包不需要修改即可正常使用
